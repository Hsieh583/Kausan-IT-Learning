**模組A：ISMS概念與ISO 27001** (2小時)
```
核心主題:

1. 資訊安全管理系統(ISMS)定義
   ├─ 人員 (Human) → 政策、角色、培訓
   ├─ 流程 (Process) → 評估、稽核、改善
   ├─ 技術 (Technology) → 控制措施實施
   └─ 持續改善 (Continuous Improvement) → PDCA循環

2. ISO 27001:2022標準架構
   ├─ 前言與介紹
   ├─ 規範要求 (Normative Requirements)
   ├─ 114項控制措施 (A5-A18)
   └─ 附錄與補充指導

3. ISO 27001 vs ISO 27002
   ├─ 27001: 認證標準 (組織層級要求)
   ├─ 27002: 實施指南 (控制措施詳細說明)
   └─ 27003: 實施指南 (組織層級)
   
4. ISMS 5大核心要素
   ├─ 背景環境 (Context)
   ├─ 領導層 (Leadership)
   ├─ 計畫 (Planning)
   ├─ 支援 (Support)
   ├─ 運作 (Operation)
   ├─ 績效評估 (Performance Evaluation)
   └─ 改善 (Improvement)
```

#### Day 2: ISMS實施與運作 (8小時)
#### Day 2: ISMS Implementation & Operation

**模組A：風險管理流程** (3小時)
```
ISO 27001風險管理架構:

步驟1: 風險評估 (Risk Assessment)
├─ 資產識別 (Identify Assets)
│   └─ IT設備、數據、應用、人員
│
├─ 威脅識別 (Identify Threats)
│   ├─ 自然災害 (地震、颱風)
│   ├─ 技術故障 (硬體故障)
│   ├─ 人為威脅 (駭客、內部員工)
│   └─ 法律風險 (合規違反)
│
├─ 弱點識別 (Identify Vulnerabilities)
│   ├─ 配置缺陷
│   ├─ 軟體漏洞
│   ├─ 人員疏漏
│   └─ 流程缺失
│
├─ 影響評估 (Impact Assessment)
│   ├─ 機密性損失 (Confidentiality)
│   ├─ 完整性損失 (Integrity)
│   ├─ 可用性損失 (Availability)
│   └─ 財務損失計算
│
├─ 可能性評估 (Likelihood Assessment)
│   └─ 低(1) / 中(2) / 高(3)
│
└─ 風險評分 = 影響 × 可能性

步驟2: 風險評估準則設定 (Risk Criteria)
├─ 可接受風險等級
├─ 風險評估方法
└─ 評估時間表

步驟3: 風險處理決策 (Risk Treatment)
├─ 風險避免 (Avoid)
│   └─ 停止高風險業務
├─ 風險減低 (Mitigate)
│   └─ 實施控制措施降低風險
├─ 風險接受 (Accept)
│   └─ 記錄風險並接受
└─ 風險轉移 (Transfer)
    └─ 購買保險或外包

步驟4: 風險處理計畫 (Treatment Plan)
├─ 選擇控制措施 (114項)
├─ 定義實施責任人
├─ 設定完成期限
├─ 分配資源與預算
└─ 制定驗收標準
```

**模組B：控制措施實施** (3小時)
```
如何實施114項ISO 27001控制:

分類法1: 按控制類型
├─ 技術控制 (Technical Controls)
│   └─ 防火牆、加密、訪問控制系統
├─ 物理控制 (Physical Controls)
│   └─ 門禁卡、監視器、機房保安
├─ 政策控制 (Policy Controls)
│   └─ 資訊安全政策、程序、指南
└─ 組織控制 (Organizational Controls)
    └─ 審批流程、職責分離、培訓

分類法2: 按優先級別
├─ 必要控制 (Mandatory)
│   └─ 法規或業務要求
├─ 高優先 (High Priority)
│   └─ 風險評估識別
├─ 中優先 (Medium Priority)
│   └─ 改善項目
└─ 低優先 (Low Priority)
    └─ 長期改善

控制實施流程 Implementation Process:
1. 選擇控制 (Selection)
   └─ 根據風險評估選擇適用控制
   
2. 詳細設計 (Detailed Design)
   └─ 定義控制的具體實施方式
   └─ 制定SOP(標準操作程序)
   
3. 實施 (Implementation)
   └─ 部署技術系統
   └─ 建立流程與程序
   └─ 培訓員工
   
4. 驗證與測試 (Verification & Testing)
   └─ 測試控制有效性
   └─ 修復發現的問題
   
5. 監控與評估 (Monitoring & Evaluation)
   └─ 定期測試
   └─ 績效指標監控
   └─ 調整與改善
```

#### Day 3: Internal Audit Planning & Preparation

**模組A：稽核方法論** (3小時)
```
ISO 27001內部稽核是什麼?
定義: 對ISMS進行的系統性、獨立性評估，
      確認組織是否符合ISO 27001要求，
      並評估ISMS有效性。

目的 Objectives:
✅ 確認ISMS符合標準要求
✅ 驗證控制措施有效性
✅ 識別改善機會
✅ 提供認證審核的證據
✅ 管理層了解ISMS狀態

稽核類型 Types of Audits:
1. 完整稽核 (Full Audit)
   └─ 覆蓋所有114項控制
   └─ 需時 3-5天
   └─ 年一次

2. 部分稽核 (Partial Audit)
   └─ 覆蓋特定控制域 (如存取控制)
   └─ 需時 1-2天
   └─ 季度一次

3. 快速檢查 (Spot Check)
   └─ 針對特定控制
   └─ 需時 2-4小時
   └─ 月度或按需

稽核原則 Audit Principles:
├─ 獨立性 (Independence)
│   └─ 稽核人員不能是被稽核部門的人
│
├─ 證據導向 (Evidence-based)
│   └─ 基於客觀證據判斷
│   └─ 記錄所有發現
│
├─ 系統性 (Systematic)
│   └─ 遵循稽核計畫
│   └─ 覆蓋所有範圍
│
├─ 保密性 (Confidentiality)
│   └─ 保護稽核發現
│   └─ 限制信息存取
│
└─ 建設性 (Constructive)
    └─ 幫助改善而非懲罰
    └─ 提供可行建議
```

│   ├─ 文件審查 (是否有新程序)

│   ├─ ✅ 有效關閉 (改善確實生效)

**不符合判定標準** (2小時)

```
判定流程 Decision Process:

Step 1: 證據確認
├─ 收集足夠證據支撐判定
├─ 向被稽核部門確認事實
├─ 考慮是否有其他解釋
└─ 只有確實的情況才判定為不符合

Step 2: 標準條款對應
├─ 明確指出違反哪一條ISO 27001條款
├─ 參考ISO 27002實施指南
└─ 確保判定有據可查

Step 3: 嚴重程度評級
├─ Critical (關鍵級)
│   定義: 直接導致ISMS失效，無法提供基本安全保護
│   判定標準:
│   - 缺乏法規要求的基本控制 (如風險評估)
│   - 缺乏ISMS管理層結構 (如無政策、無目標)
│   - 完全未實施某個關鍵域的控制
│   例: 完全未進行風險評估
│   - 完全未進行內部稽核
│   - 完全未建立事件管理流程
│   行動: 立即必須糾正，延遲可能無法通過認證審核
│
├─ Major (重大級)
│   定義: 不符合ISO 27001要求，但未完全失效
│   判定標準:
│   - 政策/程序缺失某些關鍵內容
│   - 控制措施未正確實施或有缺陷
│   - 未按規定頻率執行某流程
│   例: 政策未進行年度評審
│   - 帳號權限未定期審查
│   - 培訓紀錄不完整
│   行動: 必須在3個月內糾正，否則認證審核可能不通過
│
└─ Minor (輕微級)
    定義: 程序方面的改善空間，不影響ISMS有效性
    判定標準:
    - 文件格式或記錄不規範
    - 流程執行有延遲但已完成
    - 單個案例未遵守但不是系統性問題
    例: 某次變更申請單格式不完整
    - 某個員工培訓紀錄簽名漏簽
    - 某次備份日誌格式不標準
    行動: 建議改善，無硬性要求，但應在後續改善

Step 4: 記錄與溝通
├─ 將不符合記錄在"不符合紀錄表"
├─ 向被稽核部門主管說明
├─ 給予被稽核部門提出異議的機會 (24小時內)
├─ 如被稽核部門有異議，稽核小組評估是否維持判定
└─ 若有調整，更新紀錄
```
