# IT營運優先層級框架（L1-L5）

**版本**: 1.0  
**建檔日期**: 2026年1月23日  
**策略方向**: 營運優先（Operational-First）- 以不中斷、不爆炸為核心，逐層提升IT品質

---

## 框架總覽

```
L5: 文件與合規證書        ◄── 最後才達成的驗收層
     ↑
[證據自動化流水線]       ◄── 橋接層：規範化多源數據→自動生成登記簿
     ↑
L4: 工具與自動化          ◄── 加速交付、減少人力浪費
L3: 使用者體驗            ◄── 人用起來順，降低摩擦
L2: 防禦與監控            ◄── 出事有訊號、能快速處理
L1: 可靠性與容灾          ◄── 核心：不中斷、不爆炸（DR/HA）
```

## 第1層：可靠性與容灾（L1 - Reliability & Disaster Recovery）

### 層級定義
IT營運的**基礎層**。確保業務關鍵系統和數據不中斷、不丟失。任何一個系統不穩定都會導致營運中斷，因此這是必須首先完成的層級。

### 具體目標
| 面向 | 目標 | 驗收標準 |
|------|------|--------|
| **系統可用性** | 關鍵系統保持99.5%以上可用性 | RTO ≤ 4小時, RPO ≤ 1小時 |
| **備份保護** | 所有業務數據每日備份，異地存儲 | 備份完整率100%, 還原測試成功率≥1次/月 |
| **容灾計畫** | 完善的DR計畫與定期演練 | 年度DR演練≥2次, 演練成功率100% |
| **故障自動轉移** | HA集群配置完畢 | 自動轉移時間≤5分鐘 |

### 成熟度指標（Maturity Level）

| 等級 | 定義 | 對標業界 | 實施標誌 |
|------|------|--------|--------|
| **1 - 初始** | 有基本備份，無HA配置 | 中小企業 | 週期性手動備份存在 |
| **2 - 基礎** | 備份自動化，有簡單的容灾計畫 | 傳統企業 | 備份自動執行，紀錄完整 |
| **3 - 可靠** | HA系統部分實施，DR計畫已驗証 | 中型企業 | 關鍵系統有主備配置，年度演練通過 |
| **4 - 高可用** | 全面HA+主動DR，自動轉移成熟 | 大型企業 | 異地多活配置，轉移自動化 |
| **5 - 卓越** | 全球多地點冗餘，業務連續性零中斷 | 超大型企業 | 實時複製，跨域自動轉移 |

### 關鍵實施要點
- [ ] 盤點所有業務關鍵系統（CMS, ERP, 客戶資料庫等）
- [ ] 評估每個系統的RTO/RPO需求
- [ ] 實施自動備份機制（每日1次，異地存儲）
- [ ] 建立並測試容灾恢復程序
- [ ] 部署HA集群（至少主備配置）
- [ ] 建立故障轉移自動化（監控→檢測→轉移）
- [ ] 記錄並監控可用性指標（Monthly SLA Review）
- [ ] **[輕量化監控選項]** 對於資源受限系統（如WMS），允許以L2自動化探針替代人工彙整報告

### 對應ISO 27001控制項
| ISO控制項 | 內容 | 實施方式 |
|----------|------|--------|
| A17 | 業務連續性與災難復原管理 | DR計畫、演練紀錄、RTO/RPO定義 |
| A12.3.1 | 事件管理與隔離 | 故障偵測、自動轉移程序 |
| A8.1.4 | 備份與恢復 | 備份執行紀錄、還原測試報告 |

## 第2層：防禦與監控（L2 - Defense & Monitoring）

### 層級定義
**保護層**。系統穩定了，還要能偵測威脅、記錄事件、快速應對。「出事有訊號」→ 發現問題。「能處理」→ 有應對程序。

### 具體目標
| 面向 | 目標 | 驗收標準 |
|------|------|--------|
| **威脅偵測** | SIEM/防禦系統運作，發現異常行為 | 告警響應時間≤1小時 |
| **日誌採集** | 所有系統日誌集中管理 | 日誌完整率≥95%, 保留期≥90天 |
| **事件回應** | 事件發現→調查→修復全流程自動化 | MTTR (Mean Time To Repair) ≤ 4小時 |
| **安全監控** | 24/7監控，警報機制啟用 | 監控中心在線率≥99% |

### 成熟度指標（Maturity Level）

| 等級 | 定義 | 對標業界 | 實施標誌 |
|------|------|--------|--------|
| **1 - 初始** | 基本防火牆，無集中監控 | 小型企業 | 各系統獨立日誌存儲 |
| **2 - 基礎** | 有日誌採集，告警規則簡單 | 傳統企業 | SIEM系統已部署，基本規則生效 |
| **3 - 主動** | 多層防禦+異常檢測，事件程序完善 | 中型企業 | EDR/SIEM運作，事件紀錄齊全 |
| **4 - 進階** | 自動化告警+智能分析，威脅狩獵開展 | 大型企業 | ML異常檢測，自動化隔離, 威脅狩獵小組 |
| **5 - 卓越** | 全面的zero-trust架構+主動威脅情報 | 超大型企業 | 實時威脅情報集成，自動化應對 |

### 關鍵實施要點
- [ ] 實施SIEM系統（或ELK Stack等開源方案）
- [ ] 定義日誌採集範圍（OS, 應用, 網路, 安全設備）
- [ ] 建立告警規則（異常登入、數據外洩、惡意軟體等）
- [ ] 建立事件回應程序（檢測→分類→隔離→調查→修復）
- [ ] 配置自動隔離機制（infected hosts的自動隔離）
- [ ] 記錄MTTR指標（每月檢討）
- [ ] 培訓安全運維團隊
- [ ] **[多源數據接入標準]** 定義各類報告來源的數據接入規範（NAS日誌、Defender告警、外包月報、IT日報/月報等），明確各報告在監控體系中的處理權責與標準化格式

### 對應ISO 27001控制項
| ISO控制項 | 內容 | 實施方式 |
|----------|------|--------|
| A12.4.1 | 事件管理與報告 | 事件日誌、事件調查報告 |
| A12.6.1 | 管理信息系統的監控 | SIEM告警紀錄、監控儀表板 |
| A13.1.1 | 網路安全 | 防火牆日誌、入侵檢測告警 |
| A14.2.1 | 安全開發政策 | 開發環境的日誌監控 |

## 第3層：使用者體驗（L3 - User Experience）

### 層級定義
**便利層**。系統穩定了、能防禦了，但如果人用起來很麻煩，團隊就會繞過流程、自創工具、增加風險。目標是讓合規流程**融入自然工作流**，而非強制執行。

### 具體目標
| 面向 | 目標 | 驗收標準 |
|------|------|--------|
| **流程簡化** | 合規操作不超過3個步驟 | 使用者完成時間≤2分鐘 |
| **工具易用性** | 介面直觀，減少培訓需求 | 首次使用成功率≥80% |
| **文件與指引** | 清晰的中文指引，非術語堆砌 | 技術人員滿意度≥4/5 |
| **回饋機制** | 使用者反饋→快速改進 | 缺陷修復周期≤1週 |

### 成熟度指標（Maturity Level）

| 等級 | 定義 | 對標業界 | 實施標誌 |
|------|------|--------|--------|
| **1 - 初始** | 流程存在但複雜，需要大量指導 | 小型企業 | 紙本表單，需要IT人員協助 |
| **2 - 基礎** | 線上系統已部署，文件初步齊全 | 傳統企業 | 表單系統上線，有基本指南 |
| **3 - 友好** | UI/UX優化，流程精簡，指引完善 | 中型企業 | 自助服務率≥70%, 用戶滿意度4/5 |
| **4 - 卓越** | 無縫集成於日常工作，幾乎無感 | 大型企業 | 自動化與集成完成，主動推薦 |
| **5 - 愉悅** | 使用者主動擁抱流程，成為文化 | 超大型企業 | 自發合規參與率>90% |

### 關鍵實施要點
- [ ] 使用者訪談（了解當前pain point）
- [ ] 簡化合規操作流程（從N步→3步）
- [ ] 優化工具界面（減少專業術語）
- [ ] 製作短視頻教程（替代長文件）
- [ ] 建立反饋通道（改進優先級清單）
- [ ] 定期滿意度調查
- [ ] A/B測試新流程的接受度
- [ ] **[人工日報數位化]** 將日報從單純文字轉為結構化數據輸入（表單化、欄位化），確保人工輸入能驅動後續自動化合規流程

### 對應ISO 27001控制項
| ISO控制項 | 內容 | 實施方式 |
|----------|------|--------|
| A6.2.2 | 人力資源安全與意識 | 簡化的培訓流程、易懂指南 |
| A5.1.3 | 責任分配 | 清晰的角色定義（通俗化說明） |

## 第4層：工具與自動化（L4 - Tools & Automation）

### 層級定義
**效率層**。手動操作易出錯且浪費時間。通過自動化工具，讓重複性工作自動執行，團隊專注於高價值活動。節省時間=降低成本+減少人為錯誤。

### 具體目標
| 面向 | 目標 | 驗收標準 |
|------|------|--------|
| **自動化覆蓋率** | 重複性工作自動化 | 日常操作自動化率≥70% |
| **工作時間節省** | 減少手動操作時間 | 人力投入↓50% (vs 初期) |
| **錯誤率** | 自動化流程的缺陷率 | 自動化缺陷率≤0.1% |
| **整合度** | 工具間資料流通順暢 | 系統集成率≥80% |

### 成熟度指標（Maturity Level）

| 等級 | 定義 | 對標業界 | 實施標誌 |
|------|------|--------|--------|
| **1 - 初始** | 工具各自獨立，大量手動操作 | 小型企業 | Excel表格, 手動紀錄 |
| **2 - 基礎** | 單個工具自動化（如備份、更新） | 傳統企業 | Cron Job/Task Scheduler上線 |
| **3 - 整合** | 多工具協作，自動化流程成熟 | 中型企業 | API集成, CI/CD初步部署 |
| **4 - 智能** | 工作流自動化，決策輔助AI | 大型企業 | Workflow automation, ML預測 |
| **5 - 自適應** | 自我優化，異常自動修復 | 超大型企業 | AIOps, 自我修復系統 |

### 關鍵實施要點
- [ ] 列舉所有重複性工作（記錄在"工作清單"）
- [ ] 優先級排序（時間成本 × 頻率）
- [ ] 採購/實施自動化工具（Ansible, Terraform等）
- [ ] 建立資料整合層（API或消息隊列）
- [ ] 開發監控儀表板（自動化狀態+效率指標）
- [ ] 記錄節省的時間與成本
- [ ] 建立自動化測試框架
- [ ] **[多源數據整合標準]** 建立統一的數據接入與轉換層，支持多種報告來源（系統日誌、人工日報、第三方報告等）自動轉換為標準格式，驅動L5證據生成

### 對應ISO 27001控制項
| ISO控制項 | 內容 | 實施方式 |
|----------|------|--------|
| A14 | 系統開發與維護 | IaC (Infrastructure as Code), 配置管理 |
| A12.2.1 | 變更管理 | 自動化變更流程, 變更追蹤 |
| A12.1.2 | 事件管理 | 自動化告警→分類→分配 |

## 第4層與第5層之間：證據自動化流水線（Evidence Pipeline）

### 定義
**橋接層**。L4的自動化與L5的文件合規之間的關鍵連接。將多源、異質的運維數據自動規範化、結構化，並生成符合ISO 27001要求的登記簿與證據鏈。

### 流水線核心流程

```
多源數據 → 格式規範化 → 數據驗證 → 自動生成登記簿 → L5證據輸出
   ↓           ↓            ↓           ↓              ↓
系統日誌    標準化轉換   完整性檢查   ISO格式產出   稽核就緒文件
人工日報    欄位映射     異常標記     版本控制       證據追溯鏈
第三方報告  格式統一     數據清洗     自動歸檔       合規驗證
```

### 關鍵組成要素

| 組成 | 說明 | 實施方式 |
|------|------|--------|
| **數據採集點** | 定義所有證據來源（SIEM、備份系統、人工表單等） | 建立數據源清單與採集API |
| **格式規範化** | 將異質數據轉換為統一結構 | ETL流程、數據字典、映射規則 |
| **驗證與清洗** | 確保數據完整性、準確性 | 自動化校驗規則、異常告警 |
| **登記簿生成** | 自動產出ISO要求的記錄與登記簿 | 模板引擎、自動化腳本 |
| **追溯與審計** | 保持證據鏈完整，可回溯原始數據 | 版本控制、操作日誌 |

### 多源報告接入標準

#### 數據源分類與處理

| 報告來源 | 層級歸屬 | 接入方式 | 輸出格式 | 頻率 |
|---------|---------|---------|---------|------|
| **NAS存儲日誌** | L2監控 | 自動採集API | 標準化CSV | 每日 |
| **Defender告警** | L2防禦 | SIEM集成 | 安全事件JSON | 實時 |
| **外包商月報** | L2/L4 | 人工上傳+自動解析 | 結構化PDF→表格 | 每月 |
| **IT日報** | L3體驗 | 結構化表單輸入 | 標準化記錄 | 每日 |
| **IT月報** | L5合規 | 自動彙總生成 | ISO登記簿格式 | 每月 |
| **WMS健康報告** | L2探針 | 自動化監控（輕量化） | 健康指標JSON | 每小時 |

#### 接入標準要求

1. **格式標準化**：所有數據源必須轉換為組織定義的標準格式
2. **完整性驗證**：必填欄位檢查、數據範圍驗證
3. **時間戳規範**：統一使用ISO 8601格式（YYYY-MM-DDTHH:MM:SSZ）
4. **追溯性**：保留原始數據來源與轉換記錄
5. **權責明確**：每個數據源指定責任單位與審查頻率

### 實施路徑

- [ ] 建立數據源清單（盤點所有需要成為證據的報告）
- [ ] 設計統一數據模型（定義標準欄位與格式）
- [ ] 開發/採購ETL工具（實現自動轉換）
- [ ] 建立驗證規則庫（確保數據品質）
- [ ] 實施自動化登記簿生成（模板化輸出）
- [ ] 建立證據追溯機制（版本控制、審計日誌）
- [ ] 定期檢查與優化（每月審查流水線健康度）

### 成功指標

- 自動化證據生成覆蓋率≥80%
- 數據轉換錯誤率≤0.5%
- 證據產出時效性：當日運維→次日證據可用
- 稽核時證據完整性檢查通過率100%

## 第5層：文件與合規證書（L5 - Documentation & Compliance）

### 層級定義
**驗收層**。前4層全做好了，文件與證書自然而然產生。這不是起點，而是終點。文件與證書用來**驗證**系統運作良好，而非指導系統設計。

### 具體目標
| 面向 | 目標 | 驗收標準 |
|------|------|--------|
| **文件完整性** | 所有操作流程有對應文檔 | 文檔覆蓋率100% |
| **稽核就緒** | 內部稽核通過率 | 發現缺陷率<5% |
| **合規驗證** | ISO認證達成 | 外部稽核通過 |
| **持續改進** | 每月文件與系統改進 | 改進項≥3項/月 |

### 成熟度指標（Maturity Level）

| 等級 | 定義 | 對標業界 | 實施標誌 |
|------|------|--------|--------|
| **1 - 初始** | 文件零散，無版本管理 | 小型企業 | Word文件各處存放 |
| **2 - 基礎** | 文檔集中，有基本版本控制 | 傳統企業 | 有共享文件夾, 基本SOP存在 |
| **3 - 組織** | 文檔體系完善，易於查找 | 中型企業 | Wiki/KMS系統上線, 文檔版本清晰 |
| **4 - 卓越** | 文檔自動生成，與系統同步 | 大型企業 | 從系統自動產出文檔, 審計軌跡完整 |
| **5 - 動態** | 活文檔，實時反映系統狀態 | 超大型企業 | 文檔與系統實時同步, AI驅動更新 |

### 關鍵實施要點
- [ ] 建立文檔體系（架構、版本、歸檔）
- [ ] 實施文件管理系統（版本控制、存取管理）
- [ ] 定期更新文檔（隨系統變更同步）
- [ ] 建立內部稽核流程（月度/季度檢查）
- [ ] 準備ISO認證申請（政策+程序+証據完整）
- [ ] 建立外部稽核清單
- [ ] 培訓文檔維護責任人

### 對應ISO 27001控制項
| ISO控制項 | 內容 | 實施方式 |
|----------|------|--------|
| A5 | 組織政策與責任 | 政策文件、角色定義 |
| A6.2.1 | 資訊安全意識與培訓 | 培訓紀錄、考試結果 |
| A12.7.1 | 監督與審查 | 稽核紀錄、管理審查報告 |

## 層級間的依賴關係

```
L5 ◄── 驗證 ◄── [證據流水線] ◄── L4 ◄── 驅動效率 ◄── L3 ◄── 提升體驗 ◄── L2 ◄── 加強防禦 ◄── L1
文件      證書      自動化證據生成    工具      自動化        使用      簡便        監控       防禦        基礎
與合規              (規範化+登記簿)   自動化                                             檢測
```

**向上滑坡是危險的** ⚠️ 
- 如果L1(可靠性)不穩，L2-L5會崩塌
- 如果L2(防禦)有漏洞，L3-L5也沒用
- 如果L3(體驗)差，L4自動化會被繞過
- 如果L4(自動化)缺失，L5文件會過時
- **如果證據流水線斷裂，L5證據將無法自動產生，回到手工整理**

## 實施路線圖（建議順序）

### Phase 1: 奠定基礎（3-6個月）
- **L1**: 建立HA+DR基礎設施
- **L2**: 部署SIEM、建立日誌採集、定義多源數據接入標準
- **產出**: 可靠性報告、事件紀錄

### Phase 2: 保護與簡化（6-9個月）
- **L2**: 完善事件回應程序、培訓團隊
- **L3**: 優化用戶流程、簡化表單、實施人工日報數位化
- **產出**: 簡化版SOP、使用者反饋

### Phase 3: 提升效率（9-12個月）
- **L4**: 自動化日常運維、建立CI/CD、建立多源數據整合層
- **L3**: 集成工具、改進體驗
- **證據流水線**: 設計統一數據模型、實施ETL流程
- **產出**: 自動化清單、時間節省統計

### Phase 4: 合規驗收（12-18個月）
- **證據流水線**: 完善自動化登記簿生成、建立追溯機制
- **L5**: 整理全部證據、進行內部稽核
- **準備**: ISO認證申請
- **產出**: 完整稽核報告、ISO認證

## 常見誤區

| 誤區 | 後果 | 正確做法 |
|------|------|--------|
| 先做L5(文檔)再做L1 | 文檔與系統脫節，認證無效 | 先做L1-L4，自然產生L5 |
| 投資L4(自動化)忽視L1 | 自動化故障擴大，更多中斷 | L1穩定→L4才能發揮 |
| L3(體驗)設計忽視L2(安全) | 使用者繞過流程，安全漏洞 | 安全優先，體驗其次 |
| 一步到位所有層級 | 資源分散，全部不達標 | 逐層深化，確保當層成熟 |
| 忽視L2(防禦)專注合規 | 出事時無法回應，認證也白搭 | 防禦是底線 |
