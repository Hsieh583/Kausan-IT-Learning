---

## IT 的角色定義（開場一句）

> **IT 提供的是一組「穩定、可恢復、可擴充的數位服務」，
> 而不是零散的技術支援。**

### 核心洞察：IT 是「可高度外包/採購的職能」

**這份 Service Catalog 本質上是「採購清單」，而非「技術清單」。**

對標業界：
- **Azure / AWS**: 按分類提供的雲服務（Compute、Storage、Network、Security 等）
- **Google Workspace**: Identity、Collaboration、Security 等功能服務
- **Microsoft 365**: Productivity、Governance、Compliance 等端到端服務
- **ServiceNow / Atlassian**: IT Operations、Governance 的標準產品

**我們的決策框架不是「自己寫什麼」，而是「採購什麼」：**
1. **定義需求**：用 Service Catalog 明確業務需要什麼
2. **評估來源**：自建、雲服務、SaaS、還是外包？
3. **管理服務**：無論來源如何，用 L1-L5 統一管理品質

**結論**：IT 經理的核心職能是「選擇和組合可靠的服務」，而非「自己構建所有東西」。

---

### 我們的服務架構基於 L1-L5 營運優先層級：

- **L1**: 可靠性與容灾 → 不中斷、不爆炸
- **L2**: 防禦與監控 → 出事有訊號、能快速處理
- **L3**: 使用者體驗 → 人用起來順，降低摩擦
- **L4**: 工具與自動化 → 加速交付、減少人力
- **L5**: 文件與合規 → 驗證系統運作品質（非起點）

## 1️⃣ Identity & Access Service

**（身分與存取服務）**

**🎯 層級對應**: L2 防禦 + L3 體驗 + L5 文件  
**⚠️ 服務失敗的風險**: 權限過度/混亂 → 內部風險；體驗差 → 人工帳號滿天飛

### 服務在賣什麼

> **「讓對的人，在對的時間，存取對的系統。」**

### 服務內容

* 帳號建立 / 停用
* 權限角色管理
* 登入與驗證機制
* 離職 / 異動權限回收

### 客戶（內部）

* HR
* 各部門主管
* 全體員工

### 服務承諾（SLO 語言）

* 帳號申請完成時間
* 權限異動處理時間
* 權限錯誤事件回應時間

### 經營指標（IT 經理要看）

* 孤兒帳號數量
* 權限異常事件
* 人工介入比例（越低越好）

### 與 L1-L5 的對應

## ⚠️ 常見誤區（根據 L1-L5 框架）

| 誤區 | 後果 | 我們怎麼做 |
|------|------|---------|
| **先做合規(L5)再做基礎(L1)** | 文檔與系統脫節；認證無效 | **L1→L5 逐層深化** |
| **忽視L1可靠性，只投L4自動化** | 自動化故障擴大；多重中斷 | **L1 是底線，L4 才能發揮價值** |
| **忽視L3體驗，強制執行流程** | 使用者繞過，自創工具，增加風險 | **安全優先，體驗其次，但都重要** |
| **每個層級都要同時完成** | 資源分散，全部不達標 | **逐層深化，確保當層成熟再進入下層** |

## 📊 實施優先順序（根據 L1-L5）

### Make vs Buy 決策框架

在每個服務上，我們面臨選擇：

| 決策 | 例子 | 適用情境 |
|------|------|--------|
| **Buy (採購/SaaS)** | Microsoft 365、AWS Backup、Okta IAM | 成熟度高、競爭激烈、無差異化、快速上線需求 |
| **Build (自建)** | 特定工作流自動化、內部整合層 | 核心競爭力、高度定制、現有系統深度整合 |
| **Manage (外包)** | MSP 提供的整體運維、SIEM SOC | 團隊能力不足、成本效益、24/7 支援需求 |
| **Hybrid (混合)** | 用 Azure + 自建應用、AWS + 內部工具 | 最常見；公雲基礎設施 + 自建應用邏輯 |

**我們的策略**: L1-L3 盡量採購/外包（成熟度高、無差異化），L4-L5 可適度自建（流程優化、文檔管理）。

---

### Phase 1: 奠定基礎 (L1-L2)
**時間**: 3-6個月  
**目標**: 確保系統不中斷、出事有訊號  
**服務**: Data Protection & Recovery → Compute & Application → Security Monitoring  
**成功標誌**: 系統可用性99.5%+；MTTR≤4小時

### Phase 2: 保護與簡化 (L2-L3)
**時間**: 6-9個月  
**目標**: 防禦更完善，使用者體驗更好  
**服務**: 完善 Security Monitoring；簡化 Identity & Access + Network  
**成功標誌**: 高風險事件閉環率100%；使用者滿意度4/5+

### Phase 3: 提升效率 (L4)
**時間**: 9-12個月  
**目標**: 減少手動，加速交付  
**服務**: Productivity & Automation；Compute 部署自動化  
**成功標誌**: 自動化覆蓋率70%+；人力投入↓50%

### Phase 4: 合規驗收 (L5)
**時間**: 12-18個月  
**目標**: 整理全部證據，準備稽核  
**服務**: IT Governance & Visibility；準備 ISO 認證  
**成功標誌**: 內部稽核發現<5%；外部稽審通過
