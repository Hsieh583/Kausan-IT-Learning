### 模組1.1：IAM的三大支柱 (2小時)
### Module 1.1: Three Pillars of IAM

#### 支柱1：身份識別 (Identification)
```
定義: 確認用戶是誰

實施機制 Implementation:
├─ 用戶名稱 (Username)
│   └─ 格式: firstname.lastname (如 john.smith)
│   └─ 唯一性: 全網路唯一
│
├─ 員工ID (Employee ID)
│   └─ 由HR分配
│   └─ 永久識別符
│
├─ 電子郵件 (Email)
│   └─ john.smith@company.com
│   └─ 通訊與識別
│
└─ 生物辨識 (Biometric)
    └─ 指紋、面部識別（未來）

管理系統 Management System:
┌─────────────────────────┐
│ Okta Identity Vault     │
│ (身份中央存放庫)        │
├─────────────────────────┤
│ 存儲用戶基本訊息:       │
│ ✅ 姓名、ID、部門       │
│ ✅ 郵件、電話、地址     │
│ ✅ 經理關係              │
│ ✅ 職位與級別            │
│ ✅ 部門與群組成員資格   │
└─────────────────────────┘

生命週期 Lifecycle:
新員工 → Okta建檔 → AD同步 → 帳號啟用 → 日常使用
```

#### 支柱2：認證 (Authentication)
```
定義: 驗證用戶確實就是其聲稱的人

驗證方法 Authentication Methods:

1️⃣ 單因素認證 (Single Factor - 已過時❌)
├─ 僅密碼
│   └─ 風險: 密碼容易被盜竊
│   └─ 本公司: 已禁用
│
2️⃣ 雙因素認證 (2FA - 基本要求✅)
├─ 密碼 + 一次性密碼 (OTP)
│   ├─ 物理Token (如RSA SecurID)
│   ├─ 應用生成的碼 (如Google Authenticator)
│   ├─ SMS簡訊碼 (較弱，但可用)
│   └─ Email確認碼
│
├─ 密碼 + 安全問題
│   └─ 例: "您出生地是?"
│   └─ 缺點: 答案可能被推測
│
└─ 密碼 + 生物辨識
    ├─ Windows Hello (指紋、臉部)
    └─ 本公司試點: Mac Touch ID

3️⃣ 多因素認證 (MFA - 推薦實施🎯)
├─ 密碼 + Okta Verify應用 + 生物辨識
│   └─ Okta Verify: 推送通知確認
│   └─ 例: 用戶登入→Okta推送通知→用戶指紋確認
│
├─ 密碼 + 硬體金鑰 + PIN
│   └─ 如YubiKey (最安全)
│   └─ 本公司: 行政人員配備
│
└─ 風險評估動態認證
    └─ Okta檢測異常 → 提升認證強度
    └─ 例: 從未見過的IP登入 → 要求MFA確認

本公司實施策略:
Phase 1 (現在): 密碼 + 應用OTP
Phase 2 (Q1): 密碼 + Okta Verify應用 (推送)
Phase 3 (Q2): 密碼 + Okta Verify + 生物辨識 (可選)
```

#### 支柱3：授權 (Authorization)
```
定義: 決定用戶可以訪問什麼資源和執行什麼操作

授權模型 Authorization Models:

A. 基於角色的訪問控制 (RBAC - Role-Based Access Control)
   ├─ 概念: 用戶 → 角色 → 權限
   │
   ├─ 例如:
   │   ├─ IT管理員角色
   │   │   └─ 權限: 建立帳號、修改密碼、查看日誌
   │   ├─ 財務分析師角色
   │   │   └─ 權限: 訪問財務系統、生成報告 (但不能修改)
   │   └─ 普通員工角色
   │       └─ 權限: 訪問郵件、檔案共享、假期系統
   │
   └─ 本公司實施:
       - 定義10+個標準角色
       - Okta管理員 (IT專職)
       - 應用管理員 (各部門)
       - 用戶 (普通員工)

B. 基於屬性的訪問控制 (ABAC - Attribute-Based Access Control)
   ├─ 概念: 根據用戶屬性決定權限
   │
   ├─ 例如:
   │   ├─ IF 部門=IT AND 級別≥3 THEN 可訪問伺服器
   │   ├─ IF 位置=台北 AND 裝置=公司筆電 THEN 可訪問VPN
   │   └─ IF 職位=經理 AND 部門=人資 THEN 可訪問薪資系統
   │
   └─ 優點: 更靈活，自動調整權限

本公司授權架構:
┌──────────────────────────────┐
│ 用戶 john.smith              │
├──────────────────────────────┤
│ 屬性:                        │
│ - 部門: IT                   │
│ - 級別: 3 (資深工程師)       │
│ - 職位: 系統管理員            │
│ - 地點: 台北                  │
│ - 經理: IT部長               │
│                              │
│ 授權結果:                     │
│ ✅ 訪問Active Directory       │
│ ✅ 訪問Proxmox虛擬主機       │
│ ✅ 訪問NAS檔案共享 (IT資料夾) │
│ ❌ 訪問薪資系統               │
│ ❌ 訪問財務報表               │
│ ✅ VPN遠端連線               │
│ ✅ VPN遠端連線 (僅公司裝置)   │
└──────────────────────────────┘
```

#### 階段1：入職前 (Onboarding - Pre-Hire)

**流程步驟**：
```
時間點: 員工正式聘用日期前1週

步驟1: HR提交員工資訊 (T-7天)
├─ 所需表單: 新員工登記表
├─ 提交人: HR部門
├─ 提交至: IT帳號管理小組 + Okta管理員
├─ 必要資訊:
│   ├─ 姓名 (中英文)
│   ├─ 員工ID (HR分配)
│   ├─ 部門與單位
│   ├─ 職位與級別
│   ├─ 報告對象 (經理)
│   ├─ 開始日期
│   ├─ 電話與辦公位置
│   └─ 所需應用列表
│
└─ 文件儲存: 人資檔案 + Okta記錄

步驟2: IT審查需求 (T-6天)
├─ Okta管理員檢查應用清單
├─ 驗證應用存取的適當性
├─ 檢查與經理的授權
├─ 識別特殊需求 (如VPN、行動設備)
└─ 返回問題或確認(24小時內)

步驟3: 採購IT硬體 (T-5到T-1天)
├─ 根據職位發放:
│   ├─ 筆記本電腦 (Windows/Mac)
│   ├─ 監視器與週邊設備
│   ├─ 電話 (若需要)
│   └─ VPN金鑰或Token (若需要)
│
├─ IT部門:
│   ├─ 硬體配置 (OS、軟體、安全設定)
│   ├─ 執行安全映像
│   └─ 測試硬體功能
│
└─ 準備時間: 3-5天

步驟4: 準備員工手冊與政策 (T-3天)
├─ 資訊安全政策
├─ 可接受使用政策 (AUP)
├─ 遠端工作政策
├─ MFA設定指南
├─ 密碼管理政策
└─ 備妥於開始日期簽署

核查清單 (Checklist):
□ 員工資訊完整輸入Okta
□ 應用授權清單已驗證
□ 硬體已採購並配置
□ 公司Wi-Fi憑證已準備
□ 員工手冊已準備
□ IT入職培訓已排定
```

#### 階段2：入職當日 (Day 1 - Hire Event)

**核心目標**：在員工第一天內完成帳號啟用與基本訪問

**詳細步驟** (共1-2小時):
```
時間表 Timeline:

08:00 - HR迎接新員工
└─ HR基本入職 (HR政策、公司介紹、辦公室導覽)

09:00 - IT帳號啟用開始
├─ 參與人: IT帳號管理員 + 新員工
│
├─ Step A: 在Okta中啟用帳號 (5分鐘)
│   └─ 管理員在Okta後台點擊"啟用"
│   └─ 自動發送歡迎郵件給員工
│   └─ 郵件包含:
│       - 臨時登入連結
│       - 首次登入指南
│       - Okta Verify應用下載連結
│       - 密碼設定要求
│
├─ Step B: 員工首次登入 (10分鐘)
│   └─ 點擊歡迎郵件中的連結
│   └─ 進入密碼重置流程
│   └─ 設定強密碼 (見密碼政策)
│   └─ 系統提示設定MFA (Okta Verify)
│
├─ Step C: 設定MFA - Okta Verify (10分鐘)
│   └─ 員工下載"Okta Verify"應用 (App Store/Google Play)
│   └─ 掃描QR碼
│   └─ 應用顯示推送通知 (在員工手機上)
│   └─ 測試: 登出 → 登入 → 批准Okta推送通知
│   └─ 確認MFA運作
│
├─ Step D: 為員工分配應用 (5分鐘)
│   └─ 在Okta中將應用分配給員工
│   └─ 例如: Gmail、SharePoint、GitHub、VPN
│   └─ 應用自動展示於員工的Okta主頁
│   └─ 員工可點擊應用圖示自動登入 (SSO)
│
└─ Step E: 驗證訪問 (10分鐘)
    └─ 員工測試訪問所有應用
    └─ 檢查是否都能正常登入
    └─ 若有問題，立即解決
    └─ 例: "能否訪問Gmail?" "能否VPN連線?"

09:45 - IT培訓與簽署 (30分鐘)
├─ 簽署資訊安全政策和可接受使用政策
├─ 資安意識簡介 (10分鐘)
│   ├─ 釣魚郵件辨識
│   ├─ 物理安全 (不離開終端無人看守)
│   ├─ 密碼保護 (不共享、不貼便簽條)
│   ├─ 事件回報 (遇問題立即聯繫IT)
│
├─ VPN設定 (如適用)
│   ├─ 安裝VPN用戶端
│   ├─ 測試連線
│   └─ 保存連線詳情
│
└─ IT聯繫資訊與支援
    ├─ IT服務台電話/郵件
    ├─ 常見問題FAQ
    └─ 給予新員工"IT快速參考卡"

10:30 - IT入職完成
└─ 確認員工已:
   ✅ 設定帳號與MFA
   ✅ 訪問所有應用
   ✅ 簽署政策
   ✅ 了解支援流程

記錄 Record:
└─ 在Okta中記錄入職時間戳
└─ 發送確認郵件給HR與經理
└─ 更新IT資源追蹤表
```

#### 階段3：在職期間 (During Employment - Active)

**帳號活動管理**：
```
A. 定期訪問權限評審 (每季度一次)

評審範圍:
├─ 用戶仍在該職位? (是/否)
├─ 用戶仍在該部門? (是/否)
├─ 用戶的應用清單仍適當? (是/否)
├─ 是否有新應用需要添加? (是/否)
├─ 是否有應用需要移除? (是/否)

評審流程:
1️⃣ IT提取用戶清單 (Manager Report)
   ├─ 所有40名員工清單
   ├─ 每位員工當前訪問的應用
   └─ 每位員工的角色與部門

2️⃣ 部門經理審核 (3天內)
   ├─ 確認員工是否仍在部門
   ├─ 確認應用訪問是否適當
   ├─ 簽署"確認函"
   ├─ 報告任何應該被移除的訪問
   │
   ├─ 審核清單:
   │  □ john.smith - IT部門 - 應用: GitHub, Docker Hub, AWS
   │    ☑️ 確認: john.smith仍在IT部門，訪問適當
   │  □ jane.doe - HR部門 - 應用: Gmail, SharePoint, Payroll System
   │    ☑️ 確認: jane.doe仍在HR部門，訪問適當
   │  □ bob.wilson - 市場部門 - 應用: Salesforce, Gmail, WordPress
   │    ❌ 問題: Bob已轉至IT部門，不應再訪問Salesforce
   │
   └─ 返回給IT

3️⃣ IT執行調整
   ├─ 根據經理反饋進行修改
   ├─ 移除不需要的訪問
   ├─ 添加新的應用訪問
   ├─ 在Okta中記錄所有變更
   └─ 向受影響員工發送通知 (若訪問被移除)

4️⃣ 合規報告
   └─ 記錄評審完成日期、參與者、任何發現
   └─ 合規證據: ISO 27001 A8.5要求

B. 密碼重置 (按需)

政策:
├─ 密碼過期: 90天
├─ 密碼長度: 最少12個字符
├─ 密碼複雜度: 大寫+小寫+數字+符號
├─ 密碼歷史: 前3個密碼不能重複使用

程序:
1️⃣ 員工忘記密碼
   └─ 訪問 https://okta.company.com/login
   └─ 點擊 "忘記密碼?" 連結

2️⃣ Okta自助重置 (推薦)
   ├─ 回答安全問題 (如出生地)
   ├─ 或使用備用郵件/電話驗證
   └─ 設定新密碼

3️⃣ IT協助重置
   ├─ 員工撥打IT服務台
   ├─ IT驗證員工身份 (員工ID + 出生日期)
   ├─ IT生成臨時密碼
   ├─ 員工收到臨時密碼 (郵件)
   ├─ 員工首次登入時被強制更改臨時密碼
   └─ 員工設定自己的新密碼

C. 應用授權變更

場景1: 職務變動 (例: 從網路工程師升職為網路主管)
├─ 員工/經理向IT提交變更請求
├─ 在線表單: 人資→IT
├─ IT評估新職位所需應用
├─ 添加新應用 (如管理員工具)
├─ 可能移除舊應用 (如測試帳號)
└─ 更新Okta記錄 & 向員工確認

場景2: 臨時項目分配
├─ 經理要求: "Jane需要訪問Project-X應用，期限3個月"
├─ IT在Okta中設定臨期訪問
│   └─ 應用: Project-X
│   └─ 使用者: jane.doe
│   └─ 開始日期: 2026-02-10
│   └─ 到期日期: 2026-05-10
│   └─ 自動過期: 是 (系統自動移除訪問)
│
├─ Jane收到郵件告知
└─ 到期時自動移除訪問 (無需人工干預)

D. 設備管理與同步

場景: 員工更換筆記本電腦
├─ 流程:
│   ├─ 收回舊裝置
│   ├─ 新裝置配置 (安裝OS、應用、VPN)
│   ├─ 新裝置與Okta整合 (Device Trust)
│   │   └─ Okta可識別公司裝置 vs 個人裝置
│   │   └─ 允許訪問敏感應用: 公司裝置✅, 個人裝置❌
│   │
│   └─ 測試訪問
```

```

時間線 (T表示轉調起始日期):

└─ IT開始準備

T-7天: 計畫新應用

│   └─ 例: HR系統、薪資軟體、招聘工具

├─ 檢查舊應用是否需要移除
│   └─ IT應用: GitHub、AWS、Docker Hub (應移除)
│   └─ HR應用: Payroll、Workday (應添加)
│
└─ 發送授權申請給新部門經理簽署

T-1天: 執行變更
├─ 移除舊應用訪問
├─ 添加新應用訪問
├─ 更新Okta用戶記錄 (部門、經理)
├─ 更新AD群組成員資格
│   └─ 移除: IT部門 AD群組
│   └─ 添加: HR部門 AD群組
│
└─ 通知員工新應用列表

T+1天: 驗證
├─ 員工驗證可訪問新應用
├─ 員工驗證無法訪問舊應用
├─ IT進行確認測試
└─ 記錄在Okta中

記錄:
└─ 職務變更申請表 (簽署)
└─ 應用變更紀錄 (在Okta中)
└─ AD群組變更日誌
```

#### 階段5：離職 (Offboarding - Termination)

**離職與帳號停用流程** (最重要的安全流程):

```
時間線:

T-30天: 提前通知
├─ HR通知IT員工即將離職
├─ 通知內容:
│   ├─ 員工姓名
│   ├─ 最後工作日期
│   ├─ 離職原因
│   └─ 交接安排
│
└─ IT制訂離職計畫

T-14天: 知識轉移計畫
├─ 識別員工管理的關鍵系統與資料

#### 強制執行策略

```

第2階段 (3月 - 4月): 強制性 (Enforced)
├─ VPN訪問: 必須MFA
├─ 敏感應用: 必須MFA
│   ├─ 管理應用 (Okta, AD, AWS)
│   ├─ 財務系統
│   └─ HR系統
│
├─ 普通應用: 推薦但非必須
│   ├─ 郵件
│   ├─ SharePoint
│   └─ 其他應用
│
├─ 管理員帳號: 必須MFA (立即)
├─ 遠端工作: 必須MFA (立即)

第3階段 (4月+): 完全強制 (Mandatory)
├─ 所有應用: 必須MFA
├─ 所有員工: 必須使用MFA
├─ 例外: 僅特殊情況 (見下)
└─ 強制: 是
```

---

**例外情況與處理**：

```
例外情況 (Edge Cases):

情況1: 員工無智能手機
├─ 條件: 確認沒有個人手機或公司手機
├─ 解決方案:
│   ├─ 方案A: 配備公司手機 (如行政人員)
│   ├─ 方案B: 使用硬體Token (YubiKey - $50/個)
│   ├─ 方案C: 使用軟體OTP (Google Authenticator - 免費)
│   └─ 方案D: 使用SMS OTP (備用方案，次好)
│
├─ 流程:
│   ├─ 員工向IT提交"MFA例外請求"
│   ├─ IT評估原因
│   ├─ 若合理，安排替代方案
│   └─ 記錄例外 (合規)
│
└─ 超時限制: 最多30天

情況2: 員工出國/無網路
├─ 場景: 員工出國出差，無法使用推送通知
├─ 解決方案:
│   ├─ 備用方案: 預先生成的10個備用碼
│   │   └─ 員工可離線使用
│   │   └─ 一次一碼，用後即消
│   │
│   ├─ 或: 臨時調整 (使用密碼)
│   │   └─ IT發放"臨時訪問令牌" (24小時有效)
│   │   └─ 出差結束後恢復MFA
│   │
│   └─ 或: 使用VPN存取國外
│       └─ VPN本身已有MFA
│
├─ 流程:
│   ├─ 員工提前通知IT
│   ├─ IT生成備用碼或發放臨時令牌
│   ├─ 員工簽署"臨時例外確認"
│   └─ 一周後自動終止
│
└─ 超時限制: 7天

情況3: 遺失手機
├─ 場景: 員工手機遺失，無法接收推送通知
├─ 應急流程 (必須立即執行):
│   ├─ Step 1: 員工立即聯繫IT
│   │   └─ 時間: 遺失後1小時內
│   │
│   ├─ Step 2: IT在Okta中禁用舊裝置
│   │   └─ Okta 後台: Users > John Smith > Devices > Remove Device
│   │   └─ 舊手機的推送通知自動失效
│   │
│   ├─ Step 3: IT發放臨時訪問碼 (OTP)
│   │   └─ 有效期: 24小時
│   │   └─ 用途: 登入系統
│   │
│   ├─ Step 4: 員工購買新手機 (或使用舊手機)
│   │   └─ 員工告知IT
│   │
│   ├─ Step 5: IT重新設定MFA
│   │   └─ 掃描新QR碼
│   │   └─ 測試推送通知
│   │   └─ 確認正常後，臨時碼自動失效
│   │
│   └─ Step 6: 記錄事件
│       └─ 記錄在Okta中
│       └─ 調查是否有未授權訪問
│       └─ 密碼重置 (可選，若懷疑帳號被盜)
│
├─ 處理時間: <1小時
└─ 成本: IT人力成本，無其他成本

情況4: 承包商/臨時工
├─ 政策: 臨時工也必須使用MFA
├─ 方案:
│   ├─ 提供公司手機 (若有)
│   ├─ 或允許使用個人手機 (受限於敏感性)
│   ├─ 或使用硬體Token (如需高安全性)
│   └─ 或VPN訪問 (VPN本身有MFA)
│
├─ 期限管理:
│   ├─ 與IT合約綁定

## 第六至七天：特權帳號管理與安全
## Days 6-7: Privileged Access Management (PAM) - 4 hours

### 核心概念

**什麼是特權帳號 (Privileged Accounts)?**

```
定義: 對系統有管理權限的帳號

例如:
├─ 域名管理員 (Domain Admin) - AD完全控制
├─ 本地管理員 (Local Admin) - 電腦完全控制
├─ 資料庫管理員 (DBA) - SQL Server完全控制
├─ 系統管理員 (SysAdmin) - 伺服器與網路管理
├─ 應用管理員 - 應用系統管理
└─ Okta管理員 - 身份系統管理

為什麼特權帳號最危險?

1️⃣ 權限最大
   └─ 一旦被盜，攻擊者可控制整個系統
   └─ 可竊取所有資料、安裝後門、破壞系統

2️⃣ 攻擊目標
   └─ 駭客首先針對特權帳號
   └─ 濫用特權帳號進行橫向移動

3️⃣ 歷史風險
   └─ 許多安全事件的根本原因是特權帳號被盜

本公司風險:
⚠️ 4個域管理員帳號 (高風險)
⚠️ 15個本地管理員帳號 (中風險)
⚠️ 2個資料庫管理員 (中風險)
⚠️ 缺乏審計與監控 (未知風險)
```

**PAM最佳實踐 (PAM Best Practices)**:

```
原則1: 最小權限 (Least Privilege)
├─ 每個管理員只獲得必要的權限
├─ 例: 網路管理員不應有資料庫訪問
├─ 定期評審 (季度一次)
└─ 移除不必要的權限

原則2: 帳號隔離 (Account Segregation)
├─ 日常帳號: john.smith (普通用戶)
├─ 管理帳號: john.smith.admin (管理員)
├─ 使用日常帳號做普通工作
├─ 僅在需要時使用管理帳號
└─ 降低日常帳號被盜風險

原則3: 強化認證 (Strong Authentication)
├─ 所有特權帳號必須使用MFA
├─ 推薦: 多因素MFA (密碼 + 硬體Token)
├─ 必須: 密碼 + 應用OTP至少
└─ 避免: 僅密碼

原則4: 審計與監控 (Auditing & Monitoring)
├─ 記錄所有特權帳號的操作
├─ 實時告警: 異常活動立即通知
├─ 定期分析: 查找可疑活動
├─ 例: 是否有用戶在午夜訪問資料庫?

原則5: 密碼管理 (Password Management)
├─ 定期更改: 90天 (特權帳號) vs 180天 (普通帳號)
├─ 複雜度: 16個字符以上
├─ 專業管理: 使用密碼管理工具
├─ 不共享: 密碼不應在多人間共享
└─ 不硬編碼: 應用中不應硬編碼密碼

本公司PAM實施計畫:
Phase 1 (立即):
  ✅ 識別所有特權帳號 (4人域管 + 15人本地管 = 19人)
  ✅ 分類風險級別 (最高風險: 域管理員)
  ✅ 強制MFA (所有特權帳號)
  ✅ 更改密碼 (所有特權帳號, 16字符最小)

Phase 2 (3月):
  ✅ 建立帳號隔離政策
  ✅ 為每位管理員分配普通帳號
  ✅ 審計並限制權限
  ✅ 部署審計工具 (如Wazuh, 已部署)

Phase 3 (4月+):
  ✅ 考慮PAM工具 (如HashiCorp Vault - 開源)
  ✅ 中央密碼保管庫
  ✅ 自動轉輪 (密碼定期自動更改)
  ✅ 會話記錄 (錄製管理員活動)
```
