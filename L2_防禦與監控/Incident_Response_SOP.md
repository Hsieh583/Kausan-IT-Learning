#### 當前狀況分析 Current State Analysis

```
本公司資安事件風險評估:

🔴 高風險因素:
├─ 無正式事件回應計畫 (無SOP)
├─ 無專責事件回應小組
├─ Wazuh監控剛上線 (知識缺乏)
├─ 平均事件檢測時間: 2-4小時
├─ 平均事件處理時間: 12小時 (太長!)
├─ 無危機溝通流程
└─ 無事後評估與改善機制

數據證據 Evidence:
2025年發生的"假設"事件案例:
├─ 釣魚郵件(12起): 檢測延遲 2-6小時
├─ 誤配置存取(3起): 檢測延遲 4小時

└─ 內部威脅(1起): 檢測延遲 8小時

- 數據外洩風險 (無及時檢測)

NIST統計數據 (2024):
├─ 平均事件檢測時間: 210天 (全球平均)
├─ 平均響應時間: 72小時 (應為<4小時)

本公司目標 (2026年底):
✅ 事件檢測時間: <1小時 (vs 2-4小時)
✅ 事件響應時間: <4小時 (vs 12小時)
✅ P1事件恢復時間: <30分鐘 (vs未知)
✅ 事件回應成功率: >95% (vs 0%)
```

#### 事件回應計畫的價值 Value of IR Plan

```
有效的事件回應計畫能:

1️⃣ 加速檢測 (Faster Detection)
   ├─ 自動化監控 (Wazuh告警)
   ├─ 定義警告訊號 (明確知道什麼是異常)
   ├─ 建立告警閾值 (自動觸發告警)
   └─ 結果: 檢測時間 從4小時 → <1小時

2️⃣ 加快響應 (Faster Response)
   ├─ 預先定義的步驟 (無需討論該做什麼)
   ├─ 預先指定的角色 (明確誰做什麼)
   ├─ 現成的工具 (無需臨時採購)
   ├─ 預演練習 (已排練過，自動反應)
   └─ 結果: 響應時間 從12小時 → <4小時

3️⃣ 減少損害 (Reduce Damage)
   ├─ 快速隔離 (防止橫向移動)
   ├─ 及時恢復 (減少停機時間)
   ├─ 防止重複感染 (根除不完全)
   └─ 結果: 損害範圍 從組織級 → 系統級

4️⃣ 改善合規性 (Better Compliance)
   ├─ ISO 27001 A16事件管理要求
   ├─ 個資法 PDPA通報要求
   ├─ 完整的事件紀錄 (法律舉證)
   └─ 結果: 合規風險 從高 → 低

5️⃣ 保護聲譽 (Protect Reputation)
   ├─ 有序的危機溝通 (不會自亂陣腳)
   ├─ 快速通知客戶 (展示專業性)
   ├─ 透明的改善計畫 (恢復信任)
   └─ 結果: 客戶保留率 提升 vs 流失

案例: 2013 Target資料外洩事件
背景: 美國零售商Target遭駭，偷取4000萬信用卡
根本原因: 延遲檢測、緩慢響應
結果: 損失$1.87億美元、300起訴訟、CEO辭職

✅ 若有有效的事件回應計畫:
  ├─ 可能在2周內而非2個月後檢測
  ├─ 可能在24小時內隔離而非持續偷盜
  ├─ 損失可能從$1.87億 → $5000萬

ROI計算 (Return on Investment):
事件回應計畫成本: $50,000 (規劃+培訓+工具)
每起P1事件平均成本: $500,000 (停機+損害+法律)
若計畫能預防1起P1事件: ROI = 10倍

本公司估算:
年度事件發生率: 2-3起P1事件
計畫成本: $50k
預期節省: $500k × 1起 = $500k
Net ROI: 10倍+ ✅
```

#### 事件分類標準 Incident Classification

**本公司事件分類體系**：

```
分類維度 Classification Dimensions:

維度1: 嚴重程度 (Severity)
├─ P0 - 關鍵 (Critical) 🔴
│   ├─ 定義: 直接威脅核心業務連續性
│   ├─ 影響範圍: 全組織 / 多個部門
│   ├─ 受害者數量: >100人或>$100k損失
│   ├─ 例: 勒索軟體攻擊、主要數據外洩、全網路中斷
│   ├─ 響應時間: <30分鐘
│   ├─ 升級: 立即通知CEO/CISO
│   └─ 啟動: 危機應變小組 (7人全到場)
│
├─ P1 - 高 (High) 🟠
│   ├─ 定義: 嚴重系統損害或資料安全威脅
│   ├─ 影響範圍: 單一重要部門或系統
│   ├─ 受害者數量: 10-100人或$10k-100k損失
│   ├─ 例: 帳號被盜、釣魚郵件竊取憑證、單系統入侵
│   ├─ 響應時間: <2小時
│   ├─ 升級: 通知IT部長+CISO
│   └─ 啟動: 事件回應小組 (核心4人)
│
├─ P2 - 中 (Medium) 🟡
│   ├─ 定義: 資料安全或系統完整性風險
│   ├─ 影響範圍: 單一系統或少數用戶
│   ├─ 受害者數量: 1-10人或<$10k損失
│   ├─ 例: 密碼嘗試失敗多次、配置誤差、應用當機
│   ├─ 響應時間: <4小時
│   ├─ 升級: 通知IT部長
│   └─ 啟動: 值班IT工程師
│
└─ P3 - 低 (Low) 🟢
    ├─ 定義: 輕微安全警告或非安全相關事件
    ├─ 影響範圍: 個別用戶
    ├─ 受害者數量: 1人且無數據損失
    ├─ 例: 遺忘密碼、應用慢速、服務台通知
    ├─ 響應時間: <24小時
    ├─ 升級: 服務台ticket記錄
    └─ 啟動: 服務台人員或IT支援

維度2: 事件類型 (Incident Type)
├─ 惡意軟體 (Malware)
│   ├─ 病毒、蠕蟲、特洛伊木馬、勒索軟體
│   ├─ 技術特徵: 檔案感染、進程異常、網路連線異常
│   ├─ 檢測方式: 防毒軟體告警, 行為分析
│   └─ 回應方式: 隔離系統, 根除惡意軟體, 恢復備份
│
├─ 駭客入侵 (Unauthorized Access)
│   ├─ 帳號被盜、漏洞利用、預設密碼
│   ├─ 技術特徵: 異常登入、新增帳號、權限提升
│   ├─ 檢測方式: AD日誌, 登入失敗告警
│   └─ 回應方式: 更改密碼, 確認訪問範圍, 追查活動
│
├─ 釣魚攻擊 (Phishing)
│   ├─ 釣魚郵件、虛假網站
│   ├─ 技術特徵: 郵件含惡意連結/附件
│   ├─ 檢測方式: 郵件安全網關, 用戶舉報
│   └─ 回應方式: 隔離郵件, 教育用戶, 重置密碼
│
├─ 數據外洩 (Data Exfiltration)
│   ├─ 未授權數據傳輸、大量下載
│   ├─ 技術特徵: 異常網路流量, 檔案訪問量激增
│   ├─ 檢測方式: DLP工具, 網路監控
│   └─ 回應方式: 阻止傳輸, 確認數據內容, 通知主管機關
│
├─ 系統故障 (System Failure)
│   ├─ 硬體故障、軟體崩潰、網路中斷
│   ├─ 技術特徵: 服務無法訪問, 系統當機
│   ├─ 檢測方式: 監控告警, 用戶投訴
│   └─ 回應方式: 故障轉移, 恢復備份, 根本原因分析
│
└─ 內部威脅 (Insider Threat)
    ├─ 員工誤操作或惡意行為
    ├─ 技術特徵: 異常檔案訪問, 大量列印
    ├─ 檢測方式: 行為分析, 用戶舉報
    └─ 回應方式: 證據保存, HR通知, 法律諮詢

事件判斷流程圖:

問題1: 是否為安全相關事件?
├─ 否 → P3 (系統故障/支援)
└─ 是 → 問題2

問題2: 是否已確認發生?
├─ 否 → 可疑事件 (待驗證)
└─ 是 → 問題3

問題3: 影響多少人/系統?
├─ 1人 → 問題4 (可能P2或P3)
├─ 2-10人 → 問題4 (可能P1或P2)
├─ 11-100人 → P1 (高優先)
└─ >100人 → P0 (關鍵)

問題4: 是否涉及數據外洩風險?
├─ 否 → P2 (中等優先)
├─ 是 → P1 (高優先)

問題5: 是否導致系統停機?
├─ 否 → P1或P2 (根據數據風險)
└─ 是 → P0 (關鍵) ← 立即升級
```

### 模組2.1：事件檢測信號 (2小時)
### Module 2.1: Event Detection Signals

#### 常見事件檢測信號 Common Detection Signals

```
信號來源與優先級:

🔴 最高優先級信號 (Highest Priority):

1. 勒索軟體檔案活動 Ransomware File Activity
   ├─ 信號: 大量檔案被加密 + 副檔名突然改變
   ├─ 檢測方式: Wazuh FIM (檔案完整性監控)
   ├─ 示例日誌:
   │   ├─ File modified: C:\Users\john\Documents\file.docx → file.docx.encrypted
   │   ├─ File encryption detected: .cryptolocker, .wannacry
   │   └─ Large number of files modified in short time (100+ in 1 min)
   │
   ├─ 告警規則: if 1分鐘內超過100個檔案被修改 → P0告警
   ├─ 人工驗證: 
   │   ├─ 用戶是否有意進行大量操作?
   │   ├─ 修改的檔案副檔名是否異常?
   │   └─ 進程是否為惡意軟體 (如PsExec, PowerShell)?
   │
   ├─ 立即行動:
   │   ├─ 隔離受感染系統 (断网 + 禁用)
   │   ├─ 停止該用戶帳號 (防止擴散)
   │   ├─ 升級為P0事件
   │   └─ 啟動危機應變中心
   │
   └─ 恢復: 從備份恢復 (3-2-1-1-0策略)

2. 異常網路流量 Unusual Network Traffic
   ├─ 信號: 大量出站連線到異常IP
   ├─ 檢測方式: Wazuh網路監控 + 防火牆日誌
   ├─ 示例日誌:
   │   ├─ Outbound connection to 192.168.0.100:4444 (known C2 server)
   │   ├─ Large data transfer: 2GB to external IP in 30min
   │   └─ Repeated connection failures to port 3389 (RDP bruteforce)
   │
   ├─ 告警規則: 
   │   ├─ if 已知C2伺服器IP → P0告警
   │   ├─ if 出站超過1GB/小時 → P1告警
   │   └─ if RDP連線嘗試>20/分鐘 → P1告警
   │
   ├─ 人工驗證:
   │   ├─ 這個系統應該連到這個IP嗎?
   │   ├─ 是否為正常業務流量 (如備份、API呼叫)?
   │   └─ 流量特徵是否符合已知惡意軟體?
   │
   ├─ 立即行動:
   │   ├─ 阻止該系統對外連線 (防火牆規則)
   │   ├─ 停用該系統 (隔離)
   │   ├─ 取證分析 (記錄所有日誌)
   │   └─ 升級為P1事件
   │
   └─ 恢復: 從備份恢復或重新安裝OS

3. 異常帳號活動 Abnormal Account Activity
   ├─ 信號: 帳號在異常時間或地點登入
   ├─ 檢測方式: Okta + Wazuh AD監控
   ├─ 示例日誌:
   │   ├─ Failed login attempts: 50+ in 5 minutes (john.smith)
   │   ├─ Successful login from unusual location: Russia (john.smith)
   │   ├─ Account locked: Too many failed attempts
   │   └─ New account created by john.smith (unusual privilege)
   │
   ├─ 告警規則:
   │   ├─ if 登入失敗>10次/分鐘 → P1告警 (暴力破解)
   │   ├─ if 從未見過的地理位置登入 → P2告警 (異常登入)
   │   ├─ if 帳號在午夜12:00-3:00活躍 → P2告警 (異常時間)
   │   └─ if 新增admin帳號由普通用戶 → P1告警 (權限提升)
   │
   ├─ 人工驗證:
   │   ├─ 用戶是否出差或在不同位置?
   │   ├─ 登入嘗試失敗是否為用戶遺忘密碼?
   │   ├─ 新增帳號是否經過授權?
   │   └─ 該帳號過去是否有異常活動?
   │
   ├─ 立即行動:
   │   ├─ 如為暴力破解: 帳號自動鎖定 (Okta防護)
   │   ├─ 如為異常登入: 聯繫用戶確認
   │   ├─ 如為權限提升: 立即移除提升權限
   │   └─ 根據情況升級為P1或P2
   │
   └─ 恢復: 重置密碼、檢查帳號活動歷史

🟠 高優先級信號 (High Priority):

4. 誤配置暴露 Configuration Mistake
   ├─ 信號: AWS S3/公開共享暴露、防火牆規則錯誤
   ├─ 檢測方式: 配置審計、敵對性掃描
   ├─ 例: 公司NAS錯誤地設為"匿名訪問"
   ├─ 升級: P1 (如涉及敏感資料) 或 P2 (一般資料)
   └─ 恢復: 修復配置、重新掃描

5. 缺失日誌/日誌被篡改 Missing/Tampered Logs
   ├─ 信號: 日誌檔案被刪除、大量日誌缺失
   ├─ 檢測方式: 日誌監控告警、Syslog驗證
   ├─ 影響: 攻擊者可能在隱蓋蹤跡
   ├─ 升級: P1 (立即)
   └─ 恢復: 從備份恢復、深度取證

🟡 中優先級信號 (Medium Priority):

6. 軟體漏洞 Software Vulnerability
   ├─ 信號: 已知漏洞的應用被訪問或執行
   ├─ 檢測方式: 弱點掃描、補丁管理
   ├─ 例: CVE-2024-1234被利用跡象
   ├─ 升級: 根據漏洞嚴重程度 (CVSS)
   └─ 恢復: 打補丁、限制訪問

信號優先級決策樹:

                是否為安全事件?
                    ↙  ↘
                  是     否 → P3 (忽略)
                  ↓
            影響範圍有多大?
           ↙   ↓   ↓   ↘
        全網  部門 部分  1用戶
        ↓      ↓    ↓     ↓
       P0     P1   P1    P2
    (危機) (高優先) (高優先) (中優先)

範例場景:

場景A: Wazuh檢測到員工終端機一分鐘內加密了50個檔案
=> 檔案異常修改 + 大量 + 短時間 = P0 (勒索軟體跡象)
=> 立即隔離系統、升級危機應變中心

場景B: Okta檢測到john.smith帳號從未見過的IP登入
=> 異常登入 + 地理位置異常 = P2 (調查)
=> 聯繫john.smith確認、評估風險

場景C: AWS發現S3桶被設為"公開讀取"
=> 配置誤差 + 可能數據外洩 = P1 (立即修復)
=> 檢查日誌誰訪問過、是否外洩數據
```

### 模組2.2：事件分析與初評 (2小時)
### Module 2.2: Event Analysis & Initial Assessment

#### 事件分析步驟 Analysis Steps

```
Step 1: 驗證事件存在 (5-15分鐘)
└─ 質問: "告警一定正確嗎? 或只是誤報?"

檢查清單:
□ 告警時間戳與事件發生時間相符嗎?
□ 涉及的系統/帳號/檔案是否存在?
□ 日誌中是否有對應的原始事件?
□ 有沒有可能是監控系統故障?
□ 有沒有可能是合法但異常的活動? (如緊急備份)

例: "Wazuh告警:john.smith帳號登入失敗50次"
驗證步驟:
├─ 檢查Okta日誌: 確認是否真的有50次失敗
├─ 聯繫john.smith: 是否遺忘密碼?
├─ 檢查失敗登入來源IP: 是否為John的正常位置?
├─ 檢查時間: 是否為工作時間?
└─ 結論: 
   - 若為遺忘密碼 → 誤報 (非事件, 聯繫服務台重置)
   - 若為攻擊跡象 → 真實事件 (暴力破解, P1)

Step 2: 評估事件範圍 (15-30分鐘)
└─ 質問: "事件影響多廣泛?"

評估維度:
├─ 受影響系統數量: 1? 5? 全網?
├─ 受影響用戶數量: 1? 10? 全公司?
├─ 受影響資料敏感程度: 公開? 內部? 敏感? 機密?
├─ 受影響資料數量: 1檔案? 1GB? 100GB?
├─ 業務影響: 無? 輕微停機? 完全癱瘓?
└─ 可能的根本原因: 惡意? 誤操作? 故障?

例: "檢測到異常檔案訪問: C:\Payroll\salary.xlsx"
評估步驟:
├─ 受影響資料: 薪資檔案 (非常敏感 ⚠️)
├─ 訪問者: IT部長李四 (授權的?)
├─ 訪問時間: 23:00 (異常時間)
├─ 訪問方式: 本地檔案訪問 (vs遠端VPN)
├─ 訪問數量: 1次 (vs多次)
└─ 結論:
   - 若李四有授權查看薪資 → 可能是合法 (降級為P2)
   - 若李四無授權查看薪資 → 內部威脅 (提升為P1)

Step 3: 確定事件類型 (15-30分鐘)
└─ 質問: "這是什麼類型的事件?"

根據之前的事件信號判斷:
├─ 惡意軟體? (檔案加密、進程異常)
├─ 駭客入侵? (帳號被盜、異常訪問)
├─ 釣魚攻擊? (用戶點擊連結、認證被盜)
├─ 數據外洩? (大量下載、異常連線)
├─ 系統故障? (服務不可用)
└─ 內部威脅? (員工異常行為)

判斷依據:
├─ 技術特徵 (Technical Indicators)
├─ 行為模式 (Behavioral Pattern)
├─ 時間戳 (Temporal Pattern)
├─ 涉及資料 (Data Accessed)
└─ 使用工具 (Tools Used)

例: "多個系統檔案被加密 + 出現勒索信 + 檔案后綴.cryptolocker"
推斷: 勒索軟體攻擊 (Ransomware)
判據:
├─ 技術: 檔案加密 + 副檔名改變
├─ 行為: 廣泛的檔案操作
├─ 工具: 已知勒索軟體特徵
└─ 結論: 高度確信為勒索軟體 (P0)

Step 4: 初評與分級 (10分鐘)
└─ 質問: "這個事件有多嚴重?"

分級標準 (見前述P0-P3):
├─ 影響人數
├─ 影響系統重要性
├─ 數據敏感度
├─ 業務損失
└─ 恢復難度

填寫初評表單 Initial Assessment Form:
```

## 第四至六天：隔離、根除與恢復
## Days 4-6: Containment, Eradication & Recovery (8 hours)

### 核心目標

**模組3.1：隔離策略** (3小時)
```
隔離三層次:

層次1: 網路隔離 (Network Isolation)
- 斷開受感染系統的網路連接
- 防止橫向移動與數據外洩
- 方式: 防火牆規則 / 網路埠禁用 / VPN斷線
- 時間: <5分鐘

層次2: 系統隔離 (System Isolation)
- 停用受感染系統或帳號
- 方式: 關閉VM / 禁用帳號 / 隔離容器
- 時間: <10分鐘

層次3: 數據隔離 (Data Isolation)
- 保護未感染資料不被進一步訪問
- 方式: 備份檔案系統 / 鎖定資料庫 / 復制內容
- 時間: <30分鐘

實施檢查清單:

□ 受感染系統已識別
□ 系統已從網路斷線
□ 系統已停用
□ 涉及帳號已禁用
□ 相關數據已備份
□ 隔離狀態已驗證 (無法連線/訪問)
□ 紀錄已備存 (取證)
```

**模組3.2：根除步驟** (3小時)
```
根除目標: 移除攻擊痕跡, 確保無法重新感染

步驟:

1. 識別攻擊根源 (Root Cause Analysis)
   ├─ 惡意軟體來源?
   ├─ 入侵初始點?
   ├─ 使用的漏洞或弱點?
   └─ 防禦缺陷?

2. 移除惡意代碼 (Remove Malicious Code)
   ├─ 使用防毒軟體掃描
   ├─ 手動刪除惡意檔案
   ├─ 殺死惡意進程
   ├─ 移除註冊表項 (Windows)
   └─ 清理系統日誌

3. 修復漏洞 (Patch Vulnerabilities)
   ├─ 打安全補丁
   ├─ 升級易受攻擊的軟體
   ├─ 禁用不必要的服務
   └─ 加強防火牆規則

4. 改變憑證 (Change Credentials)
   ├─ 重置所有被入侵帳號密碼
   ├─ 更改API金鑰
   ├─ 重新簽發證書
   └─ 更新VPN憑證

5. 驗證根除 (Verify Eradication)
   ├─ 掃描系統無惡意軟體
   ├─ 檢查無異常進程
   ├─ 驗證補丁已安裝
   └─ 確認無後門
```

**模組3.3：恢復步驟** (2小時)
```
恢復目標: 將系統恢復到正常運作狀態

步驟:

1. 準備恢復環境 (Prepare Recovery Environment)
   ├─ 建立隔離的恢復網路
   ├─ 準備潔淨備份
   ├─ 驗證備份完整性
   └─ 測試恢復流程

2. 執行恢復 (Execute Recovery)
   ├─ 從備份恢復檔案系統
   ├─ 或完全重新安裝OS
   ├─ 安裝必要軟體
   ├─ 套用補丁與安全更新
   └─ 恢復配置

3. 驗證恢復 (Verify Recovery)
   ├─ 系統能否啟動?
   ├─ 應用能否運行?
   ├─ 資料是否完整?
   ├─ 網路連接是否正常?
   └─ 效能是否恢復?

4. 監控驗證 (Monitor Verification)
   ├─ 恢復後監控24小時
   ├─ 監視異常活動
   ├─ 檢查日誌異常
   └─ 確認無復發跡象

時間估算:
- P0勒索軟體: 4-8小時
- P1駭客入侵: 2-4小時
- P1釣魚攻擊: 1-2小時
- P2配置誤差: 30分鐘-1小時
```

## 第八天：危機溝通與決策
## Day 8: Crisis Communication & Decision Making (6 hours)

### 模組5.1：危機溝通計畫 (3小時)

```
溝通對象與內容:

1️⃣ 內部溝通 (Internal Communication)

對象A: 員工 (All Staff)
├─ 時機: 事件確認後立即通知
├─ 內容:
│   ├─ 事件事實 (簡單說明, 不涉及技術細節)
│   ├─ 可能影響 (您的工作如何受到影響?)
│   ├─ 應對措施 (公司已採取什麼行動?)
│   ├─ 建議 (員工應注意什麼?)
│   └─ 聯繫方式 (有疑問請聯繫...)
│
├─ 範例郵件:
│   ┌─────────────────────────────┐
│   │ 主旨: 系統維護通知          │
│   ├─────────────────────────────┤
│   │ 親愛的同仁,                 │
│   │                             │
│   │ 本公司系統今日(2/10)        │
│   │ 14:30-18:00進行緊急安全    │
│   │ 維護,期間部分應用可能      │
│   │ 無法訪問。造成不便,         │
│   │ 特此歉意。                 │
│   │                             │
│   │ 詳情請見後續通知。          │
│   │                             │
│   │ IT部門                     │
│   └─────────────────────────────┘
│
└─ 頻率: 每4小時更新一次

對象B: 管理層 (Leadership)
├─ 時機: P0事件立即, P1事件<30分鐘
├─ 內容:
│   ├─ 事件摘要 (發生了什麼?)
│   ├─ 影響評估 (會造成多大損害?)
│   ├─ 應對計畫 (如何處理?)
│   ├─ 資源需求 (需要什麼支援?)
│   └─ 風險評估 (是否涉及法律/聲譽風險?)
│
├─ 形式: 緊急會議 (電話 / 視訊)
├─ 與會者: CEO, CFO, CISO, 法務
└─ 頻率: 每小時更新

對象C: 相關部門 (Affected Departments)
├─ 時機: 立即 (涉及部門)
├─ 內容:
│   ├─ 您的部門系統狀態
│   ├─ 您可以做什麼
│   ├─ 何時會恢復
│   └─ 聯絡人訊息
│
└─ 形式: 電話 + 郵件

2️⃣ 外部溝通 (External Communication)

對象D: 客戶 (Customers)
├─ 時機: P0事件<4小時, P1事件<8小時
├─ 內容:
│   ├─ 我們遭遇了安全事件
│   ├─ 您的資料是否受到影響? (透明說明)
│   ├─ 我們採取了什麼行動?
│   ├─ 您應該做什麼? (如更改密碼)
│   └─ 持續更新計畫
│
├─ 範例:
│   ┌─────────────────────────────┐
│   │ 尊敬的客戶,                 │
│   │                             │
│   │ 2月10日, 本公司發現安全    │
│   │ 事件。我們已立即啟動應變   │
│   │ 計畫。                     │
│   │                             │
│   │ 初步調查顯示您的資料       │
│   │ 可能受到影響。我們建議    │
│   │ 您立即更改密碼。           │
│   │                             │
│   │ 詳情: https://...          │
│   │ 聯絡: security@company.com │
│   └─────────────────────────────┘
│
└─ 頻率: 至少每天1次

對象E: 監管機構 (Regulatory Bodies)
├─ 條件: 涉及個人資料外洩的P0事件
├─ 時機: 確認外洩後72小時內 (GDPR/PDPA要求)
├─ 內容:
│   ├─ 事件詳情 (何時何地何人)
│   ├─ 外洩資料類型與數量
│   ├─ 已採取的補救措施
│   ├─ 預防措施
│   └─ 聯絡方式
│
└─ 責任: 法律部門負責

溝通決策樹:

是否為P0事件?
├─ 否 → 標準溝通 (通知內部 + 相關部門)
└─ 是 → 危機溝通
    ├─ 是否涉及數據外洩?
    │   ├─ 否 → 通知客戶 (避免謠言)
    │   └─ 是 → 通知客戶 + 監管機構
    │
    ├─ 是否可能影響聲譽?
    │   ├─ 否 → 日常通知
    │   └─ 是 → PR團隊介入
    │
    └─ 是否需要媒體聲明?
        ├─ 否 → 內部溝通為主
        └─ 是 → 準備媒體聲明 (CEO審核)
```

## 附錄A：事件回應檢查清單

### 應急檢查清單 Emergency Checklist

```
事件檢測時:
□ 接收告警或通知
□ 記錄確切時間與信息來源
□ 初步判斷是否為真實事件
□ 立即通知技術領隊

事件確認時 (<15分鐘):
□ 技術領隊進行初評
□ 確定事件分級 (P0-P3)
□ 根據等級啟動回應流程
□ 通知相應決策人員

P0事件應急 (立即):
□ 啟動危機應變中心 (所有成員到場)
□ CISO接管指揮
□ 建立情況室 (即時溝通)
□ 開始實時日誌記錄
□ 指定代言人 (外部溝通)
□ 聯絡法律與PR顧問
□ 保存所有證據 (關鍵!)

隔離階段 (<30分鐘):
□ 識別受影響系統
□ 斷開網路連接
□ 停用涉及帳號
□ 備份關鍵資料 (取證)
□ 通知資料擁有者
□ 監控隔離狀態

根除階段 (1-4小時):
□ 進行根本原因分析
□ 移除惡意軟體/代碼
□ 修復漏洞
□ 更改涉及密碼
□ 驗證根除成功

恢復階段 (1-8小時):
□ 準備恢復計畫
□ 從備份恢復系統
□ 測試系統功能
□ 恢復到生產環境
□ 進行24小時監控

事後活動 (3-5天):
□ 進行事後檢討會議
□ 制定改善計畫
□ 更新事件回應計畫
□ 進行員工培訓 (吸取教訓)
□ 完成事件報告

合規活動 (根據需要):
□ 向監管機構通報 (若需)
□ 通知受影響客戶 (若需)
□ 內部審計 (記錄保留)
□ 司法機構協助 (若涉及犯罪)
```
